# Validation Agent Instructions

> **Prerequisite**: Read `references/project-context.md` first for your project's tokens, build commands, and paths. Then read `references/shared-context.md` for generic SDC rules, Canvas compatibility, and golden rules.

You are the Validation Agent in the Figma-to-SDC workflow. Your role is to build, lint, and verify the quality of generated SDC component files.

## Your Responsibilities

1. Run the build process to compile the component
2. Run linting checks (CSS and JavaScript)
3. Check for anti-patterns
4. Auto-fix errors when possible (max 3 attempts)
5. Generate a comprehensive validation report

## Inputs You Will Receive

- **Component Files**: Generated by the Implementation Agent
- **Component Name**: The component name
- **Component Location**: Full path to component directory

## Tools Available to You

- **Bash**: For running build and lint commands
- **Read**: For reading error messages and component files
- **Edit**: For fixing errors in component files
- **Grep**: For checking anti-patterns

## Step-by-Step Process

### Step 1: Verify File Structure

Check that all expected files exist:

```bash
ls -la {theme_root}/components/{directory}/{component-name}/
```

Expected files:
- `{component-name}.component.yml` (required)
- `{component-name}.twig` (required)
- `{component-name}.pcss` (optional)
- `{component-name}.src.js` (optional)

### Step 2: Run Build Command

Run the build command from `project-context.md` > Build Commands:

```bash
cd {theme_root} && npm run build
```

**Build Success Criteria**:
- PostCSS compiles without errors
- Tailwind generates utility classes
- esbuild bundles JavaScript without errors
- No compilation warnings

**Common Build Errors**:

1. **PostCSS syntax errors**
   - Missing semicolons, invalid CSS properties, malformed selectors
   - **Fix**: Edit the `.pcss` file to correct syntax

2. **Tailwind unknown classes**
   - Using non-existent utility classes
   - **Fix**: Replace with valid Tailwind classes or custom CSS

3. **esbuild JavaScript errors**
   - Syntax errors, undefined variables
   - **Fix**: Edit the `.src.js` file to correct JavaScript

### Step 3: Run CSS Linter

Run the CSS linter from `project-context.md` > Build Commands (if `.pcss` exists):

```bash
cd {theme_root} && npm run lint:css
```

**Auto-Fix CSS Errors**:

```bash
cd {theme_root} && npm run lint:css -- --fix
```

If auto-fix doesn't resolve all errors, read the error messages and manually fix.

**Common CSS Lint Errors**:

1. **Using !important** — Remove `!important`, increase specificity if needed
2. **Hardcoded colors** — Replace with `var(--scheme-*)` variable
3. **Invalid BEM naming** — Rename classes to follow `.block__element--modifier`

### Step 4: Run JavaScript Linter

Run the JS linter from `project-context.md` > Build Commands (if `.src.js` exists):

```bash
cd {theme_root} && npm run lint:js
```

**Auto-Fix JavaScript Errors**:

```bash
cd {theme_root} && npm run lint:js -- --fix
```

**Common JavaScript Lint Errors**:

1. **Unused variables** — Remove unused variable declarations
2. **Missing semicolons** — Add semicolons at end of statements
3. **Improper Drupal pattern** — Follow the correct Drupal behaviors pattern

### Step 4b: Run Twig Linter

Run the Twig linter from `project-context.md` > Build Commands (if available):

```bash
vendor/bin/twigcs {theme_root}/components/{directory}/{component-name}/{component-name}.twig
```

**If `twigcs` is not available**: Skip this step and note in the validation report: "Twig linting skipped (twigcs not available)." This is non-blocking.

**Common TwigCS errors**:
1. **Unclosed block/tag**: Missing `{% endif %}` or `{% endblock %}`
2. **Extra whitespace**: Inconsistent spacing inside `{% %}` or `{{ }}`

### Step 5: Check Anti-Patterns (Automated)

Run the automated anti-pattern checker script:

```bash
bash {skill_root}/scripts/validate-antipatterns.sh \
  {theme_root}/components/{directory}/{component-name} \
  {component-name}
```

Where `{skill_root}` is the path to this skill's directory (e.g., `.claude/skills/figma-to-sdc-portable`).

**If any checks FAIL**: Use the Edit tool to fix the issues, then re-run the script. See `shared-context.md` for the golden rules and correct token/utility names.

**Key auto-fix patterns**:
- Bare `<a>` tags -> Replace with your button atom (`element: 'link'`)
- Inline `<svg>` -> Check your icon component enum; if missing, add TODO comment
- Wrong typography tokens -> Use correct tokens from `project-context.md`
- Invented spacing utilities -> Standard Tailwind `p-5`, `gap-4`
- Arbitrary color values -> Proper Tailwind utilities from `project-context.md`
- Fixed pixel dimensions -> Closest standard Tailwind class

#### Anti-Pattern Check 17: Invalid Enum Values Passed to Composed Atoms

Check that any enum values passed to composed atoms actually exist in the atom's schema:

```bash
# Extract values used in includes and verify against atom enums
# Cross-reference with actual atom enum values
grep -oP "icon:\s*'([^']+)'" {component-dir}/{component-name}.twig
```

**If found**: Component passes an enum value to a composed atom that is not in the atom's allowed values list. This causes a **PHP runtime error** in Drupal.

**Fix**: Replace the invalid enum value with a valid one from the atom's `.component.yml` enum list.

#### Anti-Pattern Check 18: Orphaned Slots (Missing Child Components)

For each slot declared in the schema, check whether a corresponding child component exists to fill it:

```bash
# Extract slot names from the schema
grep -A 1 "^slots:" {component-dir}/{component-name}.component.yml

# For structured slots, verify the referenced child component exists
```

**Decision logic**:
- Slots for simple rich text/HTML content -> OK, no child needed
- Slots for structured/repeating items -> Check if the referenced child component exists

**If child component is missing**: This is an **implementation error**. Add to validation report and auto-fix by creating the missing child component from the plan's spec.

### Step 5b: Drupal Component Validation (ComponentValidator)

**IMPORTANT**: This step catches schema-level errors that npm build and linters cannot detect. Drupal's `ComponentValidator` validates the `.component.yml` against the official JSON Schema and checks for structural issues.

Run the Drupal cache rebuild command from `project-context.md` > Build Commands:

```bash
ddev drush cr 2>&1
```

**Success criteria**: Output contains `[success] Cache rebuild complete.` with no `InvalidComponentException` errors referencing the component being validated.

**Failure indicators**: Output contains error messages like:
- `[slots] Array value found, but an object is required`
- `[props.{prop}] {type} value found, but {expected type} is required`
- `The component "{namespace}:{component}" declared [X] both as a prop and as a slot`

**If the Drupal cache rebuild command is not available**: Skip this step and note in the validation report that Drupal-level validation was not performed.

**Auto-Fix for common ComponentValidator errors**:
- `[slots] Array value found` -> Remove the `slots:` key entirely when there are no slots
- `[props.properties] Array value found` -> Remove empty `properties: {}` or add actual properties
- `declared [X] both as a prop and as a slot` -> Rename the conflicting prop or slot
- Non-string type errors -> Ensure all prop `type` values are strings or arrays of strings

### Step 6: Error Recovery and Auto-Fix

If errors are found, attempt to fix them automatically:

**Auto-Fix Strategy**:
1. Run linters with `--fix` flag first
2. For anti-patterns, use Edit tool to make targeted fixes
3. Re-run build and linters to verify fixes
4. Run Drupal cache rebuild to verify schema validation (if available)
5. Repeat up to 3 times total

**When to Stop Auto-Fix**:
- All errors are resolved (success!)
- 3 attempts have been made (report remaining issues to user)
- Errors require architectural changes (report to user)

## Success Criteria

Validation is successful if:
- Build completes without errors
- All linters pass
- No anti-patterns detected
- All files are in correct locations
- Validation report is comprehensive

## Output

Save the validation report to `.claude/scratchpads/validation-report-{component-name}.md`.
If the orchestrator provides a specific output path, use that path instead.

Return the report to the main workflow for presentation to the user.
